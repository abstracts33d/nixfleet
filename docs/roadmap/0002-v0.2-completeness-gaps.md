# Roadmap 0002 — v0.2 RFC completeness gaps

**Status:** active remediation cycle (gap #2 in flight, additional gaps queued). **Last updated:** 2026-04-28. **Owner:** open.

## What this is

A walk through RFC-0001 / RFC-0002 / RFC-0003 against the actual
codebase, classifying each declared contract as **HONORED**,
**PARTIAL**, **IGNORED**, or **DEFERRED**. Surfaces "negative
bloat" — schema fields and wire-protocol contracts the framework
claims to honor but doesn't.

The agnosticism work (phases 1–14) was about *additive* discipline
— not shipping speculative implementations. This roadmap covers
the inverse: *subtractive* honesty about which RFC promises the
framework actually keeps.

## Roll-up

~70% of the three RFCs is implemented and exercised in production.
The core intent loop (RFC-0001 declarative → RFC-0002 reconcile →
RFC-0003 wire) is solid for the steady-state case the lab fleet
exercises. The gaps cluster in two areas:

1. **Soak timers + wave state machines.** Declared in the schema,
   never fired in code. The reconciler advances waves when all
   hosts *leave* the in-flight states, not when they've *soaked*
   for N minutes. `Wave.soak_minutes` is dead-letter.
2. **Rollback machinery + probe evaluation.** Policy fields exist
   (`onHealthFailure`, `runtime_probes`), are read for logging,
   but emit no revert actions and consult no probe results.

## High-impact gaps (worth fixing before claiming v0.2 complete)

### 1. `CheckinResponse.target` missing wave / activate / confirmWindow

**RFC-0003 §4.1** prescribes `target = {closureHash, channelRef, rollout, wave, activate.confirmWindowSecs, activate.confirmEndpoint, runtimeProbes}`. The wire struct (`crates/nixfleet-proto/src/agent_wire.rs:121-127`) carries only `closure_hash`, `channel_ref`, `evaluated_at`. The dispatch loop (`crates/nixfleet-control-plane/src/dispatch.rs:54`) computes `rollout_id` internally but doesn't populate it on the response.

**Why it matters:** agents cannot observe which rollout/wave they're part of or what confirm-window deadline applies. The CP's `rollback_timer.rs` handles deadline enforcement server-side, but the agent flying blind on the wave its target belongs to is a real visibility gap.

**Fix shape:** widen `EvaluatedTarget` proto with `Option<rollout_id>`, `Option<wave_index>`, `Option<ActivateBlock {confirmWindowSecs, confirmEndpoint}>`. Populate at dispatch. Two-side serde additions, no protocol-version bump (additive nullable fields).

**Cost:** ~1 day. Single PR per crate boundary.

### 2. Wave soak timer  *(IN PROGRESS — step 1+2 landed; step 3 pending)*

**RFC-0002 §3.1** specifies "host is `Healthy` for `soakMinutes` then transitions to `Soaked`." The reconciler advances waves when no host is in `{Dispatched, Activating, ConfirmWindow, Healthy}` (`reconciler/src/host_state.rs:70-71`), interpreting "all left in-flight" as "all soaked" — same outcome only if `Healthy` is never re-entered, which isn't a guaranteed invariant.

**Why it matters:** the soak window is the safety gate that prevents fast-burning rollouts. Without it, a host that activates → flips healthy → flips back → dies leaves the rollout believing the wave succeeded.

**Fix shape:** track `last_healthy_since: DateTime<Utc>` per (rollout, host); the reconciler advances `Healthy → Soaked` only if `now - last_healthy_since ≥ wave.soak_minutes`. Persistence in CP's SQLite (`crates/nixfleet-control-plane/src/db.rs`).

**Implementation status:**

- ✅ **Step 1 (commit `fe3baec`).** New `host_rollout_state` table (V003 migration), DB methods (`record_host_healthy` / `clear_host_healthy` / `host_soak_state_for_rollout` / `healthy_rollouts_for_host`), confirm + checkin handlers wired to record/clear the marker.
- ✅ **Step 2 (commit `4052402`).** `Rollout` widened with `last_healthy_since`; `active_rollouts_snapshot()` joins pending_confirms + host_rollout_state; `observed_projection.rs` populates `active_rollouts` from the DB; reconcile tick threads the snapshot.
- ⏳ **Step 3 (next session).** Reconciler `Healthy` arm in `host_state.rs` consults `rollout.last_healthy_since` against `wave.soak_minutes`; new `Action::SoakHost { rollout, host }` variant; CP-side action handler writes `host_state='Soaked'` on the action.

**Cost (cumulative):** ~3 days as estimated. Step 3 is ~half a day.

### 3. `onHealthFailure` rollback emission

**RFC-0002 §5.1**: `onHealthFailure ∈ {halt, rollback-and-halt}`. The reconciler (`host_state.rs:74-86`) emits `HaltRollout` when policy is set, but **doesn't generate a `RollbackHost` action** for the `rollback-and-halt` variant. The CP's `rollback_timer.rs` handles the confirm-window-timeout case but not the policy-driven case.

**Why it matters:** operator declares `rollback-and-halt`, expects auto-revert. Code logs the policy string and halts. Silent semantic gap.

**Fix shape:** add `Action::RollbackHost { host, target_ref }` variant; emit when `policy.on_health_failure == "rollback-and-halt"` and host transitions to `Failed`. Agent-side: handle the new action (re-activate prior generation).

**Cost:** ~2 days. Action enum + reconciler emission + agent activation handler.

### 4. Confirm-handler idempotency for orphan confirms  *(NEW — gap A)*

**Surfaced by:** the CP-rebuild deep-dive while landing gap #2 step 1+2 (this cycle).

The `/v1/agent/confirm` handler returns 410 Gone when no matching `pending_confirms` row exists. Per RFC-0003 §4.2 the agent then triggers local rollback. After a CP rebuild (Phase 10 / done-criterion #1 of ARCHITECTURE.md §8) the in-flight `pending_confirms` rows are gone — agents that activated successfully but hadn't yet posted confirm get rolled back unnecessarily. Costs the operator one extra reboot per affected host during the recovery cycle.

**Why it matters:** silently weakens the "destroying the CP is an outage, not a breach" claim. The fleet recovers but pays an avoidable activation churn.

**Fix shape:** when the handler would 410, cross-check the agent's `request.generation.closure_hash` against the verified `FleetResolved.hosts[hostname].closureHash`. Match → upsert a synthetic `confirmed` `pending_confirms` row + `record_host_healthy`, return 204. Mismatch → continue to 410. The mTLS-CN + closure_hash cross-check matches the same authorization invariant the positive flow already trusts.

**Cost:** ~1h, single file (`handlers.rs`). Wire-impact zero.

### 5. Agent-attested `last_confirmed_at` for soak recovery  *(NEW — gap B)*

**Surfaced by:** same deep-dive.

After a CP rebuild, `host_rollout_state.last_healthy_since` is gone for every host. Hosts that were mid-soak when the CP died restart their soak window from zero on the next confirm — a soft-state regression of up to one full `soak_minutes` per affected wave.

**Why it matters:** silently weakens the safety-gate property of the soak timer during the recovery cycle. Not a breach but a real policy-window regression.

**Fix shape:** wire-additive optional field `last_confirmed_at: Option<DateTime<Utc>>` on `CheckinRequest` (RFC-0003 §6 — additive optional fields don't bump the protocol version). Agent persists the moment its current generation was confirmed and echoes it on every checkin. CP-side projection: when a host is converged on its target closure and the CP has no `host_rollout_state` row, repopulate `last_healthy_since` from the attested timestamp, clamped to `min(now, attested)`.

**Trust model:** the agent could claim earlier-than-truth, but it has root on its own host — the soak gate is operator-policy, not a security boundary against the host. Cross-checking against `boot_id` / `uptime_secs` is available if a fleet wants stricter enforcement.

**Cost:** ~3h split across nixfleet-proto + CP projection (call it B-cp); agent-side population (B-agent) defers to whenever the agent activation loop is fleshed out (issue #2 / Magic rollback). B-cp lands idle until agents start populating the field; old agents that don't send it fall back to today's behaviour (no regression).

### 6. Signed `revocations.json` sidecar artifact  *(NEW — gap C)*

**Surfaced by:** same deep-dive — the only **security-material** gap on CP rebuild.

The `cert_revocations` table (V002) is wiped on CP rebuild. Issue #14 currently documents this as accepted debt — *"Revocation list is empty (the restarted CP does not have the history; operator must re-declare revocations)"*. That is the only piece of CP-resident state where loss-on-rebuild is a security regression rather than an operational rough edge: previously-revoked agent certs become valid again until the cert's natural expiry. With 30-day cert validity (`AGENT_CERT_VALIDITY`), the regression window is real.

**Why it matters:** turns "outage, not a breach" into "outage *and* a re-validation of revoked certificates." Strict reading of done-criterion #1 of ARCHITECTURE.md §8 fails on this state. Closing this gap is what makes the v0.2 sovereignty principle actually hold.

**Fix shape:** new signed sidecar artifact alongside `fleet.resolved.json`. Operator workflow:

1. Operator commits a revocation entry (hostname + notBefore + reason) to the fleet repo.
2. CI signs `revocations.json` with the org root key (same trust class as the CI release key, see `nixfleet-release` signing pipeline).
3. CP fetches + verifies on each reconcile tick (mirrors the `fleet.resolved.json` poll structure), replays into the in-memory + on-disk `cert_revocations` set.

Operator UX shifts from "CLI-on-CP" to "git commit" — that's a *strengthening* of the inversion-of-trust property, not a regression. The UX shift is the same as why fleet topology is git-resident.

**Cost:** ~1 day. New artifact format + signing path in `nixfleet-release`; CP-side fetch + verify + replay. Coordinates with issue #12 (Signed artifacts).

### 7. Phase 10 teardown test + ARCHITECTURE.md soft-state classification  *(NEW — gap D)*

**Tracked as:** existing issue #14 (P1). This cycle expands its scope.

Issue #14 already specifies the teardown automation (microvm.nix scenario + recovery runbook + acceptance criteria). The deep-dive added two doc deliverables that fall under the same validation cycle:

1. **Per-table soft-state classification in ARCHITECTURE.md.** Today's doc handwaves at "operational state reconstructs within one reconcile tick" — directionally true but per-table reality is more nuanced. Catalogue every CP-resident table (`token_replay`, `cert_revocations`, `pending_confirms`, `host_rollout_state`, future) with its recovery profile: soft (recoverable from agent inputs / acceptable regression) vs hard (must come from signed artifacts).
2. **Explicit framework / commercial boundary** in `docs/commercial-extensions.md` (new), referenced from ARCHITECTURE.md §"Non-goals". HA replication, real-time signed state snapshots, SLA observability, audit packages — out of scope for the open kernel by design.

Both deliverables land in the same cycle as the teardown test itself. With gaps A + B + C closed, the teardown test passes against the principle's strict reading.

**Cost (delta on top of the existing #14 scope):** ~2h of doc writing. The main #14 budget is the microvm.nix scenario, unchanged.

## What's solid (no gap)

- RFC-0001 declarative schema (mkFleet emission + JCS canonicalization) — every field round-trips through proto crate; the `freshness_window ≥ 2 × signing_interval_minutes` invariant is enforced at eval time.
- Edge predecessor blocking (RFC-0002 §4.1).
- Disruption-budget enforcement (RFC-0002 §4.2) — `budget_max()` consults all budgets, returns tightest.
- Selector algebra (RFC-0001 §3) — `tags`, `tagsAny`, `hosts`, `channel`, `all`, `not`, `and` all evaluate correctly. Pre-resolved to host lists at eval time.
- Signature verification + freshness gate (RFC-0001 §4.3, RFC-0003 §7 CP-side).
- mTLS endpoint surface (RFC-0003 §1 + §7) — every endpoint route exists, CN-binding enforced per request.
- Protocol versioning header (RFC-0003 §6) — `X-Nixfleet-Protocol: 1` middleware shipped.
- `/v1/enroll` + `/v1/agent/renew` (cert lifecycle).

## Pointers for whoever closes the gaps

- The reconciler is intentionally stateless + read-only ([`crates/nixfleet-reconciler/src/lib.rs`](../../crates/nixfleet-reconciler/src/lib.rs)). State mutations happen at the CP tier (`dispatch.rs`, `rollback_timer.rs`, DB writes). Soak-timer and rollback-emission work needs to add CP-side state-tracking, then reconciler can consult it through `Observed`.
- Wire-additive proto changes (gap #1) are safe under RFC-0003 §6 — additive optional fields don't bump the protocol version. Larger changes (e.g. agent-side identity model rework, gap §2 of RFC-0003) need a §VIII amendment.
- Tests already exist for verify-artifact, dispatch decision, and reconcile actions. Soak-timer / rollback work should add fixtures rather than re-evaluate the existing trust + selector path.
