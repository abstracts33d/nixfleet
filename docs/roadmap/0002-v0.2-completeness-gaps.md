# Roadmap 0002 — v0.2 RFC completeness gaps

**Status:** documenting; remediation deferred pending priority. **Last updated:** 2026-04-27. **Owner:** open.

## What this is

A walk through RFC-0001 / RFC-0002 / RFC-0003 against the actual
codebase, classifying each declared contract as **HONORED**,
**PARTIAL**, **IGNORED**, or **DEFERRED**. Surfaces "negative
bloat" — schema fields and wire-protocol contracts the framework
claims to honor but doesn't.

The agnosticism work (phases 1–14) was about *additive* discipline
— not shipping speculative implementations. This roadmap covers
the inverse: *subtractive* honesty about which RFC promises the
framework actually keeps.

## Roll-up

~70% of the three RFCs is implemented and exercised in production.
The core intent loop (RFC-0001 declarative → RFC-0002 reconcile →
RFC-0003 wire) is solid for the steady-state case the lab fleet
exercises. The gaps cluster in two areas:

1. **Soak timers + wave state machines.** Declared in the schema,
   never fired in code. The reconciler advances waves when all
   hosts *leave* the in-flight states, not when they've *soaked*
   for N minutes. `Wave.soak_minutes` is dead-letter.
2. **Rollback machinery + probe evaluation.** Policy fields exist
   (`onHealthFailure`, `runtime_probes`), are read for logging,
   but emit no revert actions and consult no probe results.

## High-impact gaps (worth fixing before claiming v0.2 complete)

### 1. `CheckinResponse.target` missing wave / activate / confirmWindow

**RFC-0003 §4.1** prescribes `target = {closureHash, channelRef, rollout, wave, activate.confirmWindowSecs, activate.confirmEndpoint, runtimeProbes}`. The wire struct (`crates/nixfleet-proto/src/agent_wire.rs:121-127`) carries only `closure_hash`, `channel_ref`, `evaluated_at`. The dispatch loop (`crates/nixfleet-control-plane/src/dispatch.rs:54`) computes `rollout_id` internally but doesn't populate it on the response.

**Why it matters:** agents cannot observe which rollout/wave they're part of or what confirm-window deadline applies. The CP's `rollback_timer.rs` handles deadline enforcement server-side, but the agent flying blind on the wave its target belongs to is a real visibility gap.

**Fix shape:** widen `EvaluatedTarget` proto with `Option<rollout_id>`, `Option<wave_index>`, `Option<ActivateBlock {confirmWindowSecs, confirmEndpoint}>`. Populate at dispatch. Two-side serde additions, no protocol-version bump (additive nullable fields).

**Cost:** ~1 day. Single PR per crate boundary.

### 2. Wave soak timer

**RFC-0002 §3.1** specifies "host is `Healthy` for `soakMinutes` then transitions to `Soaked`." The reconciler advances waves when no host is in `{Dispatched, Activating, ConfirmWindow, Healthy}` (`reconciler/src/host_state.rs:70-71`), interpreting "all left in-flight" as "all soaked" — same outcome only if `Healthy` is never re-entered, which isn't a guaranteed invariant.

**Why it matters:** the soak window is the safety gate that prevents fast-burning rollouts. Without it, a host that activates → flips healthy → flips back → dies leaves the rollout believing the wave succeeded.

**Fix shape:** track `last_healthy_since: DateTime<Utc>` per (rollout, host); the reconciler advances `Healthy → Soaked` only if `now - last_healthy_since ≥ wave.soak_minutes`. Persistence in CP's SQLite (`crates/nixfleet-control-plane/src/db.rs`).

**Cost:** ~3 days. Touches reconciler decision, CP host-state tracking, requires a DB migration.

### 3. `onHealthFailure` rollback emission

**RFC-0002 §5.1**: `onHealthFailure ∈ {halt, rollback-and-halt}`. The reconciler (`host_state.rs:74-86`) emits `HaltRollout` when policy is set, but **doesn't generate a `RollbackHost` action** for the `rollback-and-halt` variant. The CP's `rollback_timer.rs` handles the confirm-window-timeout case but not the policy-driven case.

**Why it matters:** operator declares `rollback-and-halt`, expects auto-revert. Code logs the policy string and halts. Silent semantic gap.

**Fix shape:** add `Action::RollbackHost { host, target_ref }` variant; emit when `policy.on_health_failure == "rollback-and-halt"` and host transitions to `Failed`. Agent-side: handle the new action (re-activate prior generation).

**Cost:** ~2 days. Action enum + reconciler emission + agent activation handler.

## What's solid (no gap)

- RFC-0001 declarative schema (mkFleet emission + JCS canonicalization) — every field round-trips through proto crate; the `freshness_window ≥ 2 × signing_interval_minutes` invariant is enforced at eval time.
- Edge predecessor blocking (RFC-0002 §4.1).
- Disruption-budget enforcement (RFC-0002 §4.2) — `budget_max()` consults all budgets, returns tightest.
- Selector algebra (RFC-0001 §3) — `tags`, `tagsAny`, `hosts`, `channel`, `all`, `not`, `and` all evaluate correctly. Pre-resolved to host lists at eval time.
- Signature verification + freshness gate (RFC-0001 §4.3, RFC-0003 §7 CP-side).
- mTLS endpoint surface (RFC-0003 §1 + §7) — every endpoint route exists, CN-binding enforced per request.
- Protocol versioning header (RFC-0003 §6) — `X-Nixfleet-Protocol: 1` middleware shipped.
- `/v1/enroll` + `/v1/agent/renew` (cert lifecycle).

## Pointers for whoever closes the gaps

- The reconciler is intentionally stateless + read-only ([`crates/nixfleet-reconciler/src/lib.rs`](../../crates/nixfleet-reconciler/src/lib.rs)). State mutations happen at the CP tier (`dispatch.rs`, `rollback_timer.rs`, DB writes). Soak-timer and rollback-emission work needs to add CP-side state-tracking, then reconciler can consult it through `Observed`.
- Wire-additive proto changes (gap #1) are safe under RFC-0003 §6 — additive optional fields don't bump the protocol version. Larger changes (e.g. agent-side identity model rework, gap §2 of RFC-0003) need a §VIII amendment.
- Tests already exist for verify-artifact, dispatch decision, and reconcile actions. Soak-timer / rollback work should add fixtures rather than re-evaluate the existing trust + selector path.

## Why this stays in roadmap rather than blocking ship

Phase 1–14 was an *agnosticism* refactor: making the framework's claims of being implementation-agnostic actually true. The negative-bloat gaps above are about a *separate* axis — RFC features that exist as schemas without code. Both axes are real, but addressing them mid-decoupling would have conflated two cleanups.

The lab fleet runs in steady state without the missing pieces because lab is a single-channel single-wave deployment with a one-host coordinator. Disruption budgets, edges, multi-wave rollouts, soak timers, and rollback-and-halt only matter at fleet scale or under failure conditions that lab doesn't currently hit.

Schedule remediation when:
- A second active-deploy fleet exists (forces multi-wave + edges).
- Probes land (RFC-0002 §6, currently Phase 7 placeholder) — opens the Healthy → Soaked → Converged path.
- Agent crate gets fleshed out beyond skeleton (currently Phase 2 read-only verify path).
