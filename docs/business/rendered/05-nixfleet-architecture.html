01 — FONDEMENTS

Les six garanties architecturales

Ce qui distingue NixFleet de tout outil de gestion d'infrastructure existant, c'est que ses propriétés de sécurité, de reproductibilité et de conformité ne sont pas des fonctionnalités ajoutées — elles sont des conséquences directes du modèle sous-jacent de NixOS.

⚛

Reproductibilité déterministe

Le même flake.nix évalué aujourd'hui ou dans 5 ans, sur n'importe quelle machine, produit un système identique bit pour bit. C'est une garantie mathématique, pas une approximation.

↩

Rollback atomique

Chaque déploiement crée une génération NixOS immuable. La transition entre deux générations est atomique — aucun état partiel n'est possible. Retour arrière : < 90 secondes.

🛡

Souveraineté absolue

Chaque composant est auto-hébergeable. Le moteur core est open source Apache 2.0. Aucune connexion à nos serveurs n'est nécessaire pour déployer ou opérer.

🔐

Sécurité structurelle

Le Nix Store est adressé par hash SHA-256 (substitution de binaires impossible). L'impermanence élimine la persistance des malwares. Le SBOM est généré automatiquement.

📋

Auditabilité complète

Chaque état de chaque machine à tout instant T est reconstituable depuis le flake.lock et l'historique Git. L'audit NIS2 est un sous-produit de l'architecture.

✈

Déployabilité air-gap

L'architecture ne suppose aucune connectivité. Les fermetures Nix sont exportables, signées, et importables dans un réseau isolé. Certifiable pour les environnements classifiés.

Principe fondateur : Dans NixFleet, votre flotte entière est la valeur de retour d'une fonction. Mêmes entrées → même sortie. Toujours. L'infrastructure n'est pas décrite comme du code — elle est du code fonctionnel pur, évalué, et dont le résultat est cryptographiquement vérifié.

03 — COUCHE 1

Dépôt de configuration

Un dépôt Git unique décrit l'intégralité de la flotte. Un flake.nix à la racine compose les configurations machine depuis des rôles, des overlays, et des paramètres hôte. flake.lock épingle toutes les dépendances à leur hash exact — la reproductibilité est garantie par ce seul fichier.

# Structure recommandée du dépôt de flotte
flake.nix                  # Entrypoint — définit nixosConfigurations
flake.lock                 # Pin cryptographique de toutes les entrées

roles/
  base.nix               # SSH, firewall, durcissement systemd, audit
  serveur.nix            # Rôle headless
  poste-travail.nix      # Rôle développeur (GUI, outils)
  edge.nix               # Rôle IoT/embarqué

hotes/
  prod-web-01.nix        # Overrides + hardware-configuration.nix
  prod-db-01.nix

secrets/                   # Chiffrés via sops-nix (commités dans Git)
  prod-web-01.yaml

modules/
  nixfleet-agent.nix     # Service systemd de l'agent
  nis2-audit.nix         # Persistance logs conformité NIS2
  durcissement.nix       # NoNewPrivileges, ProtectSystem, etc.
  impermanence.nix       # ZFS/BTRFS + opt-in persistence

Le flake NixFleet importe ces composants et expose un point d'entrée unifié. L'opérateur ne manipule jamais directement les configurations machine — il commit dans Git, et l'infrastructure converge.

04 — COUCHE 2

Cache binaire Attic

La construction de l'intégralité d'une fermeture NixOS depuis les sources prendrait des heures par machine. Attic — écrit en Rust, successeur de Cachix, compatible S3 — sert de cache binaire central. Une fois un path construit et signé, toute la flotte le télécharge en secondes.

Signing & vérification

Chaque store path est signé avec la clé privée de l'organisation. Les machines rejettent silencieusement tout path non signé ou dont le hash ne correspond pas.

Hébergement EU obligatoire

Pour les tiers Pro et Enterprise, le cache est hébergé sur S3/MinIO EU (Scaleway, OVHcloud). Les données ne quittent jamais le territoire européen.

Réduction drastique du temps de déploiement

Sans cache : 8–15 minutes pour un déploiement de 100 machines. Avec Attic (94% de hit rate après la première semaine) : 40–90 secondes.

05 — COUCHE 3

nixfleet-agent

Un binaire Rust statique, injecté dans chaque machine via nixfleet.nixosModules.agent. Responsabilité unique : maintenir la machine dans l'état de génération désigné par le plan de contrôle, avec rollback automatique si les health checks échouent.

# Injecté automatiquement — hardening systemd maximal
systemd.services.nixfleet-agent = {
  description  = "NixFleet Fleet Agent";
  after        = [ "network.target" ];
  serviceConfig = {
    ExecStart            = "${nixfleet-agent}/bin/nixfleet-agent";
    Restart              = "on-failure";
    NoNewPrivileges      = true;
    ProtectSystem        = "strict";
    PrivateTmp           = true;
    PrivateNetwork       = false;   # doit parler au plan de contrôle
    ReadWritePaths       = [ "/nix/var/nix" "/var/lib/nixfleet" ];
    AmbientCapabilities  = "";
    CapabilityBoundingSet= "";      # aucune capability
  };
};
ParamètreDéfautDescription
poll_interval5mFréquence de vérification d'une nouvelle génération
grpc_streamtrueConnexion gRPC persistante (push d'urgence)
max_deploy_time15mDélai avant rollback automatique si health check échoue
canary_percent5% de la flotte pour déploiement canary initial
reboot_window"02:00-04:00"Fenêtre de maintenance pour reboots (impermanence)
health_check_cmdsystemd unitsCommande personnalisée de vérification post-déploiement
06 — COUCHE 4

nixfleet-server — Plan de contrôle auto-hébergé

Un binaire Rust unique (nixfleet-server) qui expose une API REST, une interface gRPC pour les agents, et sert le dashboard SvelteKit. Déployé comme un service NixOS — configuré depuis le même flake que le reste de la flotte.

services.nixfleet-server = {
  enable        = true;
  domain        = "fleet.monentreprise.fr";
  port          = 8080;
  grpcPort      = 9090;
  database      = "postgresql";
  gitBackend    = "https://git.interne.fr/flotte";
  ssoProvider   = "saml";
  atticCache    = "https://cache.interne.fr";
  auditLog      = true;    # Append-only, conforme NIS2
  sbomExport    = true;    # CycloneDX / SPDX auto-générés
  tlsCertFile   = config.sops.secrets.tls-cert.path;
};

07 — SECRETS

Gestion des secrets

Community / Pro — agenix

Chiffrement asymétrique avec des clés age. Secrets chiffrés commités dans Git. Chaque machine déchiffre uniquement les secrets qui lui sont destinés. Rotation = commit Git signé.

Enterprise — OpenBao (fork EUPL de Vault)

Secrets dynamiques, rotation automatique, audit trail complet. OpenBao est recommandé pour les organisations soumises à NIS2 ou DORA — souveraineté EU assurée, licence EUPL.

sops.secrets.database_password = {
  sopsFile    = ./secrets/prod-web-01.yaml;
  owner       = "postgres";
  mode        = "0400";
  restartUnits = [ "postgresql.service" ];
};
services.postgresql.passwordFile =
  config.sops.secrets.database_password.path;
08 — ROLLBACK ATOMIQUE

Générations NixOS — L'arme opérationnelle

Chaque nixos-rebuild switch crée une génération : un snapshot immuable et complet de l'état du système. Passer d'une génération à l'autre est atomique — il n'existe pas d'état intermédiaire. Si un déploiement échoue, l'agent revient automatiquement à la génération précédente en moins de 90 secondes.

OpérationCommandeDurée typique
Déploiement flotte 100 machinesnixfleet deploy --fleet=prod1–3 min (cache chaud)
Rollback d'une machinenixfleet rollback prod-web-01< 30 sec
Rollback de toute la flottenixfleet rollback --fleet=prod --gen 421–3 min
Rollback automatique (health check KO)Déclenché par l'agent< 90 sec
Déploiement canary (5% → 100%)nixfleet deploy --canarySelon config
Avantage NIS2 : L'obligation de reprise après incident en moins de 24 heures (NIS2, Article 21) est satisfaite par défaut. Un rollback de flotte entière prend moins de 3 minutes — pas 24 heures.
09 — SÉCURITÉ

Défense en profondeur par couches

🔑

mTLS entre toutes les couches

Agent ↔ plan de contrôle en mTLS. Chaque machine reçoit un certificat client unique lors de l'enrôlement, signé par la PKI NixFleet interne.

🧮

Store adressé par contenu (SHA-256)

Chaque dérivation est identifiée par le hash de ses entrées. Substituer un binaire modifié est physiquement impossible — le hash ne correspond pas.

💥

Impermanence (optionnel, recommandé)

Effacement de tout état non déclaré au reboot : malwares, backdoors, dérive de config. Activable par machine. Non obligatoire pour adopter NixFleet.

📦

SBOM automatique (sbomnix)

SBOM CycloneDX/SPDX généré depuis flake.lock à chaque déploiement. Conforme aux exigences de traçabilité de la chaîne d'approvisionnement NIS2.

📋

Log d'audit append-only

Chaque déploiement, rollback, connexion, modification : log structuré append-only. Exportable vers SIEM (Splunk, Elastic). Requis SOC2/ISO27001/NIS2.

🔒

Chiffrement LUKS intégral

Chiffrement LUKS activé par défaut en tier Enterprise et Souverain. Clés gérées via TPM2 ou HSM. Support matériel certifiable.

10 — AIR-GAP

Déploiement dans des réseaux isolés

# 1. Machine de build (côté internet)
nixfleet build --flake ./flotte --export-closure fermeture.tar.zst

# 2. Vérification de signature
nixfleet verify-closure fermeture.tar.zst --key signing.pub

# 3. Transfert via media approuvé (USB chiffré, diode optique)
scp fermeture.tar.zst hote-transfert:/media/entrant/

# 4. Import dans le store Nix isolé (aucun trafic sortant)
nixfleet import-closure /media/entrant/fermeture.tar.zst

# 5. Déploiement sans internet
nixfleet deploy --fleet=classifie --offline

Cache MinIO local

Le cache Attic est auto-hébergé sur MinIO dans le périmètre isolé. Aucun trafic entrant/sortant pendant les déploiements.

Git miroir (Forgejo)

Le dépôt de config est synchronisé vers un Forgejo interne via le même processus de transfert que les fermetures.

Diode de données

Protocole asynchrone supportant les hardware one-way. Les fermetures entrent ; les métriques et logs sortent vers le SOC.
